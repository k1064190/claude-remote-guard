Statusline Indicator

Why

Users wanted the guard state (armed vs bypassed) visible in the status line, but Claude Code renders a single statusLine.command and gives plugins no way to register, stack, or append to it. The old README told users to hand-edit their own status-line script — brittle, and impossible for anyone running a status-line plugin that owns the whole line.

What

/guard-statusline [install|uninstall|status]:

  • install composes the guard line on top of whatever status line you already have and repoints statusLine at a wrapper (idempotent; re-run to refresh)
  • uninstall restores the original statusLine object verbatim
  • status reports the current wiring and guard state
  • renders "🔒 guard: armed" / "🔓 guard: R|W|RW bypass" on its own final line

How

Two scripts. statusline-guard.sh is the render-time wrapper: it runs the recorded inner command with stdin unchanged (eval, in a set +u subshell so optional unset vars don't abort it), prints its output, then appends the guard line read from the bypass-read|write marker files (honoring flag contents/expiry, read-only). guard-statusline-setup.sh does the one-time wiring with jq: saves the current statusLine.command (string) and full .statusLine object (for exact restore), merges over the existing statusLine (preserving siblings like padding), copies the wrapper to a stable path (~/.claude/remote-guard/statusline.sh) so the settings entry survives plugin version bumps, and writes settings atomically (temp+rename, resolving symlinked dotfiles portably). Command arg parsed with first-token expansion (no glob). require_valid_settings refuses malformed JSON; warn_if_shadowed walks up to the nearest ancestor .claude/ to warn when a project/local statusLine would shadow the user-level one. Auto-install on plugin install is impossible (no supported hook edits settings.json), so the command is the minimum required user action; it is disable-model-invocation.

Retrospective

Composition (record inner → wrap → restore) is what makes this conflict-free with any status line; the copy-to-stable-path step keeps settings.json valid across plugin updates. Each review round caught real issues, including regressions in earlier fixes — the adversarial multi-reviewer pass paid off.
