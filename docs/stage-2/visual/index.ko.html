/unguard 우회를 시간 제한적이고 필요하면 영구적으로 만들어, 세션 전체를 초기화하는 대신 가드가 스스로 다시 활성화되도록 했습니다.

Unguard TTL Persist

왜

사용자가 자신이 무엇을 하는지 잘 알고 있는 상황에서 세션마다 /unguard를 다시 실행하는 것은 번거로웠습니다. 그리고 SessionStart의 일괄 rm -f 리셋은 재시작할 때마다 모든 우회를 초기화해버렸습니다. 사용자는 이 리셋 자체를 명령에서 제어할 수 있기를, 가능하다면 시간 제한을 두어 가드가 스스로 다시 활성화되기를 원했습니다.

무엇을

/unguard <read|write|all> [30m|2h|45s|N|persist]:

  • 지속 시간 → 우회가 재시작 후에도 유지되며 자동 만료(최대 24시간; 단순 숫자는 분 단위); 세션 도중 만료되면 바로 다음 가드 대상 명령에서 가드가 다시 활성화됨
  • persist → /unguard off할 때까지 유지
  • 인자 없음 → 기존과 동일한 세션 범위 토글(100% 하위 호환)
  • status가 남은 시간을 표시

어떻게

플래그 파일의 내용이 모드를 인코딩합니다: 비어 있으면 세션, 숫자면 만료 epoch, persist면 영구, 그 외 값이면 비활성(절대 과소보호하지 않음). SessionStart 훅은 일괄 rm -f에서 guard-remote-session-reset.sh로 바뀌었으며, 이 스크립트는 세션 범위이거나 만료되었거나 인식할 수 없는 플래그만 정리합니다. guard-remote-ops.sh에는 명령마다 동일한 만료 검사를 수행하는 flag_active()가 추가되었습니다. /unguard 슬래시 명령은 모든 인자를 한 단어로 전달하므로, 토글 스크립트가 $*를 다시 분리합니다.

리뷰 루프

code-reviewer-pro 서브에이전트가 확인 후 읽기 레이스 컨디션을 지적했습니다 — [ -f ] 확인과 cat 사이에 플래그가 삭제되면 ''로 읽혀 "세션 우회가 활성 상태"로 잘못 판단됩니다. flag_active()와 state()에서 수정 (cat … || return inactive, 커밋 c1c61a7). 반려한 항목: 세션 리셋 스크립트에도 동일한 레이스가 있지만 결과적으로는 여전히 올바름(파일이 결국 삭제됨), 그리고 범위 밖의 추가 인자 사용법 경고. Codex PR 리뷰는 PR에서 실행됩니다.

  • scripts/guard-remote-ops.sh:83-101 — 만료를 인식하는 우회 검사 flag_active()
  • scripts/guard-remote-session-reset.sh — 새로운 SessionStart 정리 스크립트

회고
