1단계

기존 CLI 화이트리스트를 빠져나가던 컨테이너 레지스트리 push/login을 잡아내도록 가드 훅을 확장했습니다.

Container Registry Guard

왜

가드 훅은 고정된 CLI 화이트리스트(gcloud, gsutil, bq, aws, az, kubectl, ssh, scp, terraform)에 대해서만 동작했습니다. 레지스트리로의 docker push는 그 목록에 없었기 때문에 승인 프롬프트 없이 이미지가 push될 수 있었고, 이는 Doctor Cho가 정확히 겪은 빈틈입니다. 레지스트리 push는 원격 인프라에 도달하므로 마찬가지로 가드되어야 합니다.

무엇을

docker / podman / nerdctl (그리고 하이픈이 들어간 docker-compose / podman-compose v1 바이너리)이 이제 가드되지만, 레지스트리로 향하는 작업에 한해서만이며 write로 분류됩니다:

  • push / publish — image push, manifest push, compose push/publish 포함
  • buildx 레지스트리 익스포터 / 캐시 write — --push 약식 표기와 --output/-o/--cache-to에 붙는 type=registry / push=true 형태(읽기 전용인 --cache-from type=registry 캐시 import는 의도적으로 가드하지 않음)
  • buildx imagetools create — (멀티아키텍처) 매니페스트를 레지스트리에 게시
  • login / logout

건드리지 않은 것(Doctor Cho의 범위 결정에 따라 조용히 통과): pull, search, 그리고 모든 로컬 작업 (build, run, ps, images, logs, …). 결정에 따라 범위 밖: git push와 패키지 퍼블리싱(helm/npm/twine). /unguard write 우회는 다른 write 등급 작업과 동일하게 이들을 다스리며, /unguard read 단독으로는 그렇지 않습니다.

어떻게

(CLI만 보는 remote_re와 달리) 서브커맨드를 인식하는 매칭이라서, 로컬 docker는 프롬프트 없이 유지됩니다:

  • container_re — 명령 위치의 CLI, 그다음 (image/manifest/compose 같은 중간 단어를 선택적으로 지나) 동사 push|publish|login|logout. 해당 작업은 [^;|&]*로 같은 명령 세그먼트 안에 묶이며, 후행 경계 [[:space:];|&)<>]|$ 덕분에 구분자, ), 또는 리다이렉션(< >)이 뒤따라도 끝에 오는 동사가 매칭됩니다.
  • container_pushflag_re — buildx의 --push 약식 표기(--push, --push=true|1, 단 --push=false는 제외)로, 서브커맨드가 아니라 플래그입니다.
  • container_exporter_re — --output/-o/--cache-to에 묶인 레지스트리 익스포터 / 캐시-write 형태(type=registry, push=true)로, 읽기 전용인 --cache-from type=registry 캐시 import가 write로 오분류되지 않도록 합니다.
  • container_imagetools_re — buildx imagetools create(매니페스트 게시).
  • scan이 셸이 처리하는 방식대로 bash 줄 연속(백슬래시-개행)을 접고, 탭을 공백으로, 나머지 개행을 bash가 명령 구분자로 취급하는 대로 ;로 바꿉니다 — 그 덕분에 여러 줄로 나뉜 docker buildx build … --push는 매칭되면서, 독립된 줄의 로컬 docker build .는 조용히 지나갑니다.

탐지는 과보호 쪽으로 치우칩니다(엉뚱한 echo push 인자가 프롬프트를 띄울 수 있음). 가장 중요한 원칙이 절대 과소보호하지 않는 것이기 때문입니다.

알려진 한계(수용하고 문서화함): 기존 remote_re와 일관되게, 가드는 중첩된 셸(bash -c "docker push …"), 백틱/$() 문자열, 인용된 셸 메타문자(동사 앞 인용된 인자 안의 ;/|/&), 또는 절대 경로 호출(/usr/bin/docker push) 내부를 들여다보지 못합니다. 이를 강화하려면 컨테이너 작업에만 국한하지 않고 훅 전체에 걸쳐 일관되게(별도의 변경으로) 따옴표/셸을 인식하는 파싱을 해야 합니다.

회고

서브커맨드 인식 방식 덕분에 로컬 docker는 잡음 없이 유지하면서 push 빈틈을 막았습니다. 적대적 리뷰 패스는 제값을 했습니다: 세 번의 리뷰 패스(로컬 + 서브에이전트 둘)에 더해 Codex GitHub 봇까지, 각자가 해피패스 테스트로는 잡히지 않던 진짜 조용한 우회를 잡아냈습니다 — 레지스트리 write는 표기 방식이 많습니다(--push, 익스포터 플래그, compose publish). 앞으로 가져갈 것: 원격 write 동사의 모든 표기를 빠짐없이 열거하고, 매처가 한 줄에 두 토큰을 필요로 할 때는 줄 연속을 앞단에서 미리 접을 것.
