컨테이너 레지스트리 push/login 가드.

Container Registry Guard

왜

가드 훅은 고정된 CLI 화이트리스트(gcloud, gsutil, bq, aws, az, kubectl, ssh, scp, terraform)에서만 동작했다. 레지스트리로의 docker push는 그 목록에 없었기 때문에, 승인 프롬프트 없이 이미지가 푸시되었다 — Doctor Cho가 정확히 마주친 빈틈이다. 레지스트리 푸시도 원격 인프라에 도달하므로 반드시 가드해야 한다.

무엇을

docker / podman / nerdctl(그리고 하이픈이 들어간 docker-compose / podman-compose v1 바이너리)는 이제 가드되지만, 레지스트리로 향하는 작업에 한해서만이며 write로 분류된다:

  • push — image push, manifest push, compose push, buildx --push 포함
  • login / logout

그대로 둔 것(Doctor Cho의 범위 설정에 따라 조용히 통과): pull, search, 그리고 모든 로컬 작업(build, run, ps, images, logs, …). 결정에 따라 범위 밖: git push와 패키지 배포(helm/npm/twine).

/unguard write 우회는 다른 write 클래스 작업과 동일하게 이들을 관장하며, /unguard read만으로는 그렇지 않다.

어떻게

서브커맨드 인식 매칭(CLI만 보는 remote_re와 달리)이라 로컬 docker는 프롬프트 없이 유지된다:

  • container_re — 명령 위치의 CLI, 그다음(선택적으로 image/manifest/compose 같은 중간 단어를 지나) push|login|logout 동사. 해당 작업은 [^;|&]*로 같은 명령 세그먼트에 묶이며, 후행 경계 [[:space:];|&)]|$는 구분자나 )가 뒤따라도 종단 동사가 매칭되도록 한다.
  • container_pushflag_re — buildx --push 플래그. 서브커맨드가 아니라 플래그이므로 별도로 매칭한다.
  • 줄바꿈/탭은 공백으로 평탄화(스캔)되므로, 줄 연속으로 이어진 명령(예: 여러 줄에 걸친 docker buildx build … --push)도 한 줄로 보인다.

탐지는 과보호 쪽으로 편향된다(엉뚱한 echo push 인자가 프롬프트를 띄울 수 있다). 가장 중요한 원칙이 절대 과소보호하지 않는 것이기 때문이다.

알려진 한계(수용 및 문서화): 기존 remote_re와 일관되게, 가드는 중첩 셸(bash -c "docker push …"), 백틱/$() 문자열, 절대 경로 호출(/usr/bin/docker push)의 내부를 보지 못한다. 이를 강화하려면 컨테이너 작업만이 아니라 훅 전체에 걸쳐 일관되게 적용해야 하며(별도 변경), 이번 범위에는 포함되지 않는다.

리뷰 루프: code-reviewer-pro 서브에이전트는 critical 0 / warning 0 / 문서 전용 제안 2건을 찾았다. codex(gpt-5.5, high)는 실제 critical 과소보호를 발견했다 — 마지막 토큰인 동사 뒤에 공백 없이 ;/&/|/)가 바로 따라오는 경우를 놓쳤으며, 후행 경계를 넓혀 수정했다. antigravity(Gemini 3.1 Pro High)는 범위 내의 현실적인 누락 2건 — 여러 줄 명령과 docker-compose v1 바이너리 — 을 찾았고 둘 다 수정했다. 모든 수정에 회귀 테스트를 추가했으며, 전체 스위트 통과(43/43).

회고

서브커맨드 인식 방식은 로컬 docker를 잡음 없이 유지하면서 push 빈틈을 메웠다. 적대적 리뷰 패스는 제값을 했다: codex는 진짜 조용한 우회를 잡았고, antigravity는 해피 패스 테스트가 놓친 현실적인 누락 2건을 잡았다. 앞으로 가져갈 것: 매처가 한 줄에서 두 토큰을 필요로 할 때는 줄바꿈을 먼저 정규화하라.
