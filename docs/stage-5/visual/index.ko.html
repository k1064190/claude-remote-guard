조사 전용, 코드 변경 없음: 이 플러그인을 Codex CLI에 대해 측정한 결과 — 설치는 깔끔하게 되지만 아무것도 보호하지 못합니다 — 그래서 README.md는 이 플러그인이 Claude Code 전용임을 분명히 밝힙니다.

Codex CLI Compatibility

왜

Doctor Cho가 remote-guard가 Codex CLI에서 동작하는지 물었습니다. 이것은 평범한 이식성 질문보다 더 중요합니다: 이 플러그인의 존재 이유 자체가 조용한 우회를 막는 것이므로, "설치는 되지만 보호하지는 않는" 상태는 최악의 결과이며 — 아예 설치하지 않는 것보다 나쁩니다.

무엇을

Codex CLI 0.144.3을 대상으로, 일회용 CODEX_HOME에서 테스트했습니다(실제 ~/.codex/config.toml은 이후에 그대로임을 확인했습니다).

좋은 소식 — Codex는 Claude Code 플러그인 형식을 상당 부분 채택했습니다.

  • codex plugin marketplace add <repo>가 우리의 .claude-plugin/marketplace.json을 읽습니다.
  • codex plugin add remote-guard@cho-plugins가 성공하며, 플러그인을 $CODEX_HOME/plugins/cache/cho-plugins/remote-guard/0.3.0에 캐시하고 config.toml에 [plugins."remote-guard@cho-plugins"] enabled = true를 씁니다.
  • Codex는 동일한 훅 이벤트 이름을 가집니다: PreToolUse, PostToolUse, SessionStart, PreCompact, PostCompact, SubagentStart, SubagentStop.
  • Codex는 동일한 훅 와이어 포맷을 사용합니다: 입력은 tool_name / tool_input을 담고, 출력은 hookSpecificOutput / hookEventName / permissionDecision / permissionDecisionReason / systemMessage / continue를 인식합니다.

나쁜 소식 — 설치는 되지만 아무것도 보호하지 않습니다. 서로 독립적인 세 가지 고장이 있으며, 그중 하나만 있어도 치명적입니다:

  1. 매처 불일치. hooks/hooks.json은 Bash에 매칭합니다. Codex의 셸 도구 이름은 shell이고, 사용 중인 모델/설정(gpt-5.6-sol, 코드 모드)에서 실제 호출은 exec라는 이름의 custom_tool_call이었습니다. Bash는 결코 매칭되지 않으므로 훅은 아예 실행되지 않습니다.
  2. 페이로드 형태 불일치. Codex는 tool_input.command에 명령 문자열을 전달하지 않았습니다. 관측된 도구 호출은 명령을 JavaScript 스니펫 안에 담고 있었습니다: 
    {"type":"custom_tool_call","name":"exec",
 "input":"const r = await tools.exec_command({\"cmd\":\"echo HELLO_PROBE\",\"login\":false,...});"}
    guard-remote-ops.sh는 jq -r '.tool_input.command // empty'를 수행 → 빈 값 → exit 0 → 명령이 보호 없이 실행됩니다. 이 형태는 모델/기능에 따라 달라진다는 점(코드 모드 대 평범한 shell 도구)에 유의해야 하며, 따라서 어떤 파서든 한 가지 이상의 형태를 처리해야 합니다.
  3. 훅 신뢰(hook trust). Codex는 지속되는 훅 신뢰(trusted_hash)를 요구합니다. 이것이 없으면 훅은 조용히 건너뛰어집니다 — 오류도, 경고도 없습니다. config.toml에 등록한 프로브 훅은 발화하지 않았고 그에 대해 아무 말도 하지 않았습니다. 탈출구는 --dangerously-bypass-hook-trust이며, 우리는 의도적으로 사용하지 않았습니다.

별개로, /unguard는 Codex에서 아예 동작할 수 없습니다: 이 명령은 Claude Code의 ! bash 실행 줄과 AskUserQuestion 도구에 의존하는 프롬프트 파일인데, Codex에는 둘 다 없습니다.

순 효과: Codex에서는 plugin add가 성공을 보고하고, 어떤 오류도 표면화되지 않으며, aws … / docker push …가 아무 확인 없이 실행됩니다.

어떻게

재현 방법:

export CODEX_HOME=$(mktemp -d ~/codex-test.XXXX)   # NOT /tmp — codex refuses PATH aliases there
cp ~/.codex/auth.json "$CODEX_HOME/"
codex plugin marketplace add /path/to/claude-remote-guard
codex plugin add remote-guard@cho-plugins          # succeeds
codex exec --sandbox read-only --skip-git-repo-check "Run exactly this shell command and nothing else: echo PROBE"
# then read the tool call that was actually issued:
python3 -c 'import json,sys
for l in open(sys.argv[1]):
    o=json.loads(l); p=o.get("payload",{})
    if p.get("type")=="custom_tool_call": print(json.dumps(p)[:400])' \
  "$CODEX_HOME"/sessions/*/*/*/rollout-*.jsonl

Codex의 훅 설정은 hooks.json이 아니라 config.toml에 있습니다: 

[[hooks.PreToolUse]]
matcher = "shell"

[[hooks.PreToolUse.hooks]]
type = "command"          # variants seen in the binary: command | prompt | agent
command = "/abs/path/to/hook.sh"

미해결 질문

Codex가 전달하는 정확한 PreToolUse 페이로드는 아직 검증되지 않았습니다. 바이너리에서 필드 이름(tool_name, tool_input)은 알아냈지만, shell / exec에 대해 tool_input이 무엇을 담는지는 모릅니다. 프로브 훅은 훅 신뢰 때문에 발화하지 않았고, 이를 확인하려면 신뢰 프롬프트를 수락하는 대화형 Codex 세션이거나 --dangerously-bypass-hook-trust가 필요합니다.

이것부터 하십시오. 아래의 모든 것이 그 답에 달려 있으며, 여기서 추측하는 것이야말로 조용한 우회가 출시되는 바로 그 경로입니다.

필요한 작업

  1. 평범한 shell 호출과 코드 모드 exec 호출 양쪽에 대해 실제 PreToolUse 페이로드를 포착합니다(위 미해결 질문 참고).
  2. Codex가 만들어낼 수 있는 모든 형태에서 명령을 추출합니다 — 최소한 tool_input.command(문자열), argv 배열, 그리고 코드 모드의 JS tools.exec_command({"cmd": "…"}). 인식되지 않는 형태에서는 안전하게 실패(fail closed): 명령을 추출할 수 없으면 통과시키지 말고 확인을 요청합니다. 이는 guard-remote-ops.sh의 현재 fail-open 자세를 뒤집는 것으로, 그 자세는 매칭되지 않음이 정말로 "셸 명령이 아님"을 뜻할 때만 안전합니다.
  3. Codex의 도구 이름(shell, exec)에 훅을 등록합니다 — 두 번째 매처를 두거나 Codex 전용 config.toml 블록을 둡니다.
  4. Codex용 /unguard를 대체합니다: AskUserQuestion도, ! 줄도 없습니다. 인자 전용 명령(또는 사용자가 직접 실행하는 평범한 스크립트)이 현실적인 선택지입니다. 4단계의 대화형 설정은 이식되지 않습니다.
  5. 훅 신뢰 단계를 문서화합니다. 신뢰되지 않은 훅은 조용히 실패하면서 정상 동작하는 훅과 똑같아 보이기 때문입니다.
  6. 하나의 guard-remote-ops.sh가 두 호스트를 모두 담당할지, 아니면 Codex에 별도의 진입점을 줄지 결정합니다. 호스트를 감지하는 스크립트 하나로 두면 분류 로직(가치 있는 부분)이 한곳에 유지됩니다.

코드 위치

  • scripts/guard-remote-ops.sh — jq -r '.tool_input.command // empty'와 fail-open exit 0, 이 두 줄이 Codex에서 깨집니다
  • hooks/hooks.json — matcher: "Bash"
  • commands/unguard.md — ! 줄 + AskUserQuestion(Claude Code 전용)
  • README.md — 이식성 참고 사항
  • AGENTS.md — 이 작업을 이어받을 에이전트를 위한 레포 컨텍스트

리뷰 루프

발견 사항은 코드 diff가 아니라 직접 측정으로 얻은 것이므로, 코드에 대한 리뷰어 서브에이전트는 실행하지 않았습니다. 위의 주장들은 각각 관측된 명령 출력으로 뒷받침되며, 명시적으로 검증되지 않은 한 가지는 단정하는 대신 "미해결 질문"에 따로 밝혀 두었습니다.

회고

놀라웠던 것은 Codex가 얼마나 가까운가였습니다 — 같은 이벤트 이름, 같은 와이어 포맷, .claude-plugin/을 읽는다는 점 — 그리고 바로 그 점이 위험합니다. 거의-호환은 깔끔한 plugin add와 오류 0건을 만들어내지만 가드는 완전히 부재합니다. 앞으로의 어떤 이식 작업에든 주는 교훈: 설치 성공은 보호의 증거가 아니다. 훅이 발화한다는 것을 증명한 뒤에 믿으십시오.
